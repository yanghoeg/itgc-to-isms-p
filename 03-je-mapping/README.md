# 03-je-mapping

저널엔트리(JE, Journal Entry) 분석·검토 → ISMS-P / ITGC / 회계감사 매핑.

JE는 ITAC(응용 통제) 성격과 결산감사(부정 위험 검토) 성격이 동시에 있다. 02-itac-mapping의 결산 프로세스에 묻기엔 무게가 다르고, 외부감사인의 부정 위험 절차(회계감사기준서 240)와 직결되는 영역이라 별도 카테고리로 분리한다.

> 작성 진행 중. 본격 매핑은 ITGC 영역(`01-itgc-mapping/`) 일정 수준 채워진 후. 본 README는 영역 정의·축·관계 설정 단계.

## 1. JE testing이란

JE testing = **회계 분개(전표) 데이터 자체를 모집단 단위로 분석해서 부정·오류 위험을 식별하는 절차**.

- 외부감사인 의무 절차 (회계감사기준서 240 부정 위험, 회계감사기준서 330 위험 평가 대응)
- 회사 자체 통제(ITAC)와 별개로 **감사인이 직접 모집단 분석**
- 분석 도구 = ACL·IDEA·SQL·Python·Excel pivot. 최근 = 회계법인 자체 분석 플랫폼·AI 보조 분석 확대

**전형적 JE 분석 룰 (예시)**:
- 비정상 시간대 입력 (야간·주말·결산 직전·직후)
- 비정상 입력자 (관리자·시스템 계정·비전형 사용자)
- 비정상 거래상대 (계정과목 조합 빈도 낮은 패턴)
- 비정상 금액 (Benford's Law 일탈, round number 패턴)
- 마감 후 입력·역분개·수정 패턴
- 키워드 매칭 (반제·정정·임시 등)
- 미승인 또는 자동 승인 전표

## 2. JE의 양면성 — ITAC + 결산감사

### ITAC 성격
- ERP 안의 자동통제: 마감 후 전표 입력 차단, 승인 routing, 분개 권한 분리(SoD), 자동 분개 로직, IFRS/K-IFRS 자동 환산
- ITGC가 effective여야 ITAC 신뢰 가능 (변경관리·접근통제·로그 보존)
- → 본 레포에서는 `02-itac-mapping/processes/closing-결산.md`에서 다룸

### 결산감사 성격
- 외부감사인이 모집단 추출해서 직접 분석
- 회사 통제(ITAC)와 무관하게 부정·오류 위험 식별
- 감사 절차 자체. 회사가 통제 아무리 잘 짜도 감사인은 별도로 검증
- → 본 카테고리(`03-je-mapping/`)에서 다룸

**양면성 의미**: ITAC 검증으로 자동통제 신뢰 확보 + 결산감사 절차로 잔존 위험 검증. **둘 다 필요. 어느 하나로 대체 X.**

## 3. KISA ISMS-P와의 관계

**KISA ISMS-P 안내서에 JE testing 직접 다룬 내용 = 없음.** 검색 결과:
- 저널·분개·전표·총계정원장·일계표 = 0건
- 회계·결산·재무 = 정보 분류 예시 1건만

**왜 없는가**: ISMS-P = 정보보호 + 개인정보 보호 인증. 회계 처리 자체는 영역 X.

**다만 ISMS-P 인증기준 중 JE 관련 통제와 인접한 항목**:

| ISMS-P 인증기준 | JE 영역 매핑 |
|---|---|
| 2.6.3 응용프로그램 접근 | ERP 전표 입력·승인·수정 권한 분리 (SoD) |
| 2.6.4 데이터베이스 접근 | 회계 DB 직접 조작 차단 (DB로 직접 INSERT/UPDATE 우회 차단) |
| 2.9.4 로그 및 접속기록 | JE 입력·수정·삭제·승인 로그 보존 (외부감사 자료) |
| 2.8.6 운영환경 이관 | 자동 분개 로직·결산 routine 변경 통제 |

직접 매핑 X. 인접 영역 매핑만.

## 4. 회계감사기준서 영역

JE testing의 1차 근거:

- **회계감사기준서 240** (재무제표감사에서의 부정에 관한 감사인의 책임)
  - 부정 위험 식별 의무
  - JE 및 기타 조정사항 검토 의무 명시
- **회계감사기준서 330** (평가된 위험에 대한 감사인의 대응)
  - 위험 평가 대응으로 JE testing 절차 설계
- **PCAOB AS 2110** (Identifying and Assessing Risks of Material Misstatement) — 미국 SOX 대상
- **회계감사기준서 315** (위험 평가) — IT 환경 이해 안에서 JE 처리 시스템 이해

## 5. 본 레포에서 JE를 다루는 방식

작성 우선순위 (ITGC·ITAC 일정 수준 채워진 후 본격 시작):

1. **JE 분석 룰 카탈로그** — 전형적 룰 + 한국 비금융 환경에서 자주 발견되는 패턴
2. **ERP별 JE 데이터 추출 가이드** — SAP·Oracle EBS·더존·영림원·자체 ERP 등 추출 query·테이블 위치
3. **JE 데이터 정합성 검증 절차** — IPE(Information Produced by Entity) 검증과 연결
4. **ITGC·ITAC와의 연결 매핑** — JE testing 결과를 ITGC·ITAC 결함과 연결 분석
5. **ISMS-P 인증기준 매핑** — 위 §3 표 확장
6. **결함 사례 (익명화)** — 한국 비금융 회계감사에서 마주친 JE 패턴

## 7. JE 분석 자문 / 외주 용역

JE testing·분석은 회계법인 IT 감사인 자문의 핵심 영역 중 하나입니다. 본 레포 작성자는 ITGC·ITAC 자문과 함께 JE 분석 영역도 자문 가능합니다.

- **JE 분석 룰 설계** — 회사 ERP 환경·업종 특성에 맞춘 분석 룰 카탈로그 설계
- **JE 데이터 추출·검증** — ERP별 추출 query·IPE 검증 절차 설계
- **외부감사 JE testing 대응** — 회사 측에서 외부감사인 JE 분석 대응 자료 준비
- **부정 위험 분석 보조** — 회계감사기준서 240 영역 내부 점검

상세 자문 범위·연락 방법은 [`README.md`](../README.md)의 "ITGC · ITAC 자문 / 외주 용역" 섹션 참조.
