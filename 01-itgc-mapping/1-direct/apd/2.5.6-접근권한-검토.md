# 2.5.6 접근권한 검토

## 1. ISMS 원본

### 인증기준
정보시스템과 개인정보 및 중요정보에 접근하는 **사용자 계정의 등록·이용·삭제 및 접근권한의 부여·변경·삭제 이력을 남기고 주기적으로 검토**하여 적정성 여부를 점검하여야 한다.

### 주요 확인사항 (KISA 안내서)
1. 사용자 계정 및 접근권한 **생성·등록·부여·이용·변경·말소 등의 이력**을 남기고 있는가?
2. 적정성 **검토 기준, 검토주체, 검토방법, 주기 등을 수립**하여 정기적 검토를 이행하고 있는가?
3. 검토 결과 **과다 부여, 절차 미준수, 오·남용** 등 문제 발견 시 **조치 절차**를 수립·이행하고 있는가?

### 관련 법규
- 개인정보 보호법 제29조 (안전조치의무)
- 개인정보의 안전성 확보조치 기준 제5조 (접근권한 관리)

### 계정·접근권한 이력 항목 (KISA 안내서, 책임추적성 확보)

| 구분 | 기록 항목 |
|---|---|
| 신청정보 | 신청자/대리신청자, 신청일시, 신청목적, 사용기간 |
| 승인정보 | 승인자, 승인 또는 거부 여부, 사유 및 일시 |
| 등록정보 | 등록자, 등록일, 등록방법 (결재시스템 연동·수작업 등록 등) |
| 권한정보 | 대상 시스템명, 권한명, 권한 내역 |

### 보관 기간
- **개인정보처리자: 접근권한 기록 최소 3년 보관** (안전성 확보조치 기준)
- **점검**: 안전성 확보조치 기준 제8조 = 접속기록 **월 1회 이상 점검** 의무. SIEM·rsyslog 같은 자동화 의무 X. **수기 점검 (체크리스트 + 월별 보고서)도 가능**

### 검토 주기 (KISA 안내서)
- KISA 안내서 명시: **"최소 분기 1회 이상 권고"**
- 권고 표현 (강제 X). 회사 위험평가 결과로 자율 조정 가능
- 업계 관행: 분기 1회가 표준 (대부분 회사), 반기 1회는 일부, 월 1회는 금융권 일부

### 적정성 검토 항목 (KISA 안내서)
- 공식적인 절차에 따른 접근권한 부여 여부
- 접근권한 분류체계의 **업무목적·보안정책 부합** 여부
- 접근권한 승인자의 적절성
- **직무변경 시 기존 권한 회수 + 신규 업무 권한 적절 부여** 여부
- **업무 목적 외 과도한 접근권한** 부여 여부
- 특수권한 부여·변경·발급 현황·적정성 (2.5.5와 연결)
- **협력업체 등 외부자 계정·권한 적정성**
- 접근권한 신청·승인 내역과 **실제 권한 부여 일치 여부**
- **장기 미접속자 계정 현황 + 삭제·잠금** 여부
- **휴직·퇴직 시 지체 없이 계정·권한 회수** 여부 (2.5.1과 연결)

### 조치 절차 (KISA 안내서)
- 소명요청·원인분석·보완대책·보고체계
- 변경 적용된 권한에 대해 사용자·관련자 통지
- 반복 시 근본 원인 분석·재발방지 대책

### 결함사례 (KISA 안내서)
- **사례 1**: 검토 방법·점검주기·보고체계·오남용 기준 등이 지침에 구체적으로 정의 X → **정기 검토 미수행**
- **사례 2**: 장기 미사용자 계정 잠금·삭제 정책 있으나 **6개월 이상 미접속 계정이 활성** (검토 충실 X)
- **사례 3**: 검토 시 권한 과다 부여·오남용 의심 발견됐으나 **상세조사·내부보고 등 후속조치 X**

## 3. IT감사에서는 이렇게

### 3-A. 한국 비금융 ITGC 실무 표준 (실제 검증 범위)

접근권한 검토 = **회계감사 ITGC에서 가장 무겁게 보는 영역.** 외부감사인이 결함 가장 자주 발견하는 영역 중 하나. 권한 부여(2.5.1·2.5.5) → 사용 → **검토 사이클이 작동하는지**가 통제 본질.

| ISMS 요구 | ITGC 테스트 절차 | 모집단 | 샘플 |
|---|---|---|---|
| 1. 검토 정책 수립 | 검토 기준·주체·방법·주기·보고체계 정의된 정책 문서 (entity-level) | 정책 1건 | 1건 walkthrough |
| 2. 검토 사이클 실행 | 검토 회의록·결재 서명·검토 결과 보고서 | 1년 검토 사이클 (분기 4회·반기 2회) | **100%** (전 사이클) |
| 3. 검토 모집단 정합성 | 검토 대상 list (시스템·계정·권한) 추출 = 시스템 실제 list와 일치 | 모집단 추출 | 100% (rule-based 대조) |
| 4. 검토 결과 시정 조치 | 검토 결과 발견된 권한 과다·부적정 → 회수·삭제 이력 추적 | 발견 건 전체 | 100% |
| 5. 휴직·퇴직자 권한 회수 | HR 변동 list vs 시스템 계정 list 대조 + 적시 회수 사례 | HR 변동 전체 | 100% (rule-based) |
| 6. 장기 미접속자 처리 | 회사 정책 기준(6개월·1년 등) 미접속 계정 list + 잠금·삭제 이력 | 미접속 계정 전체 | 100% |
| 7. 외부자 계정·권한 검토 | 외부자 계정 list + 부여 사유 + 종료 후 회수 검증 (2.5.5와 묶음) | 외부자 전체 | 100% |
| 8. 이력 보관 | 부여·변경·삭제 이력 보관 (3년+) | 이력 전체 | 샘플 + 보관 기간 검증 |

### 3-B. ISMS-P가 추가로 요구하지만 ITGC 실무 범위 밖

| ISMS 요구 | 절차 | 어디 영역 |
|---|---|---|
| 검토 결과 사용자 통지 | 권한 변경 시 본인 통지 절차 | ISMS-P 인증심사 |
| 권한 분류체계의 업무목적 부합 | 분류체계 자체 적정성 검증 | ISMS-P 인증심사 (회사 자체 검증 영역) |
| 권한 오·남용 패턴 분석 | UEBA·이상 사용 분석 | **보안 영역** (SIEM·SOC) |

## 4. Q&A

### Q1. 검토 주기 — KISA 분기 1회 권고가 강제인가?
A. **권고 표현. 강제 X. 다만 분기 1회가 사실상 표준. 회사 위험평가로 자율 조정 가능.**
- KISA 안내서 원문: "검토 절차 수립 (**최소 분기 1회 이상 권고**)"
- "권고"이지 "강제" X. 회사 위험평가 결과로 반기·연 1회로 조정 가능 (단 충분한 보완 근거 필요)
- **업계 관행**:
  - 분기 1회 = 대부분 회사 디폴트
  - 반기 1회 = 일부 비금융 (위험 낮은 시스템)
  - 월 1회 = 금융권 특수권한·고위험 시스템
- **결함 판단**:
  - 검토 사이클 자체 X → 결함 (KISA 결함사례 1번)
  - 회사 정책 = 분기인데 실제 = 검토 X → 결함
  - 회사 정책 = 반기 + 위험평가 근거 + 일관 적용 → OK
  - 정책 + 사이클 + 결과 보고서까지 갖춘 게 통제 본질

### Q2. 검토 주체 — IT팀? 보안팀? 현업 부서장?
A. **현업 부서장이 1순위. IT팀·보안팀은 검토 지원·결과 취합 역할. 현업이 권한 적정성 판단 가능한 유일한 주체.**
- **현업 부서장 검토 (표준)**:
  - 본인 부서원의 권한이 업무에 필요한지 판단 가능 = 현업뿐
  - IT팀·보안팀은 "권한 분포 현황"만 보지 "이 사람이 이 권한 필요한지"는 모름
  - 회사가 IT팀 단독 검토만 하면 = 형식적 검토 = 결함 위험
- **IT팀·보안팀 역할**:
  - 검토 대상 list 추출·제공
  - 검토 결과 취합·보고
  - 시정 조치 시스템 반영
- **검증 절차**:
  - 검토 회의록·서명자 확인 (현업 부서장 서명이 있는지)
  - 검토 대상 list가 현업이 판단 가능한 형태로 제공됐는지 (시스템명·권한명·내역 포함)
- **결함 판단**:
  - IT팀 단독 검토 + 현업 부서장 서명 X → 결함 (형식적 검토)
  - 검토 list가 너무 기술적 (예: SQL role 이름만)이라 현업이 판단 불가 → 결함
  - 현업 부서장 서명 + 시정 조치까지 → OK

### Q3. 검토에서 권한 과다 발견 — 전부 회수 = 비현실적?
A. **회수가 디폴트. 다만 업무 영향 평가 + 단계적 회수 + 소명 절차로 현실 적용.**
- **표준 절차**:
  - 검토 결과 권한 과다 발견 → 사용자·관리자에게 **소명 요청**
  - 소명 후: ① 업무상 필요 입증되면 권한 유지 + 사유 기록 ② 입증 X면 권한 회수
  - 회수 시 사용자·관련자 통지
- **현실 함정**:
  - "지금 다 쓰는 권한"이라 회수 못 함 → 검토가 형식적이 됨 (KISA 결함사례 3번)
  - 회수 시 업무 영향 큼 → 단계적 회수 (1단계 사용 모니터링 강화 → 2단계 일부 회수 → 3단계 전면 회수)
- **검증 절차**:
  - 검토 → 발견 건 → 소명 요청 → 결과 → 회수·유지 결정 → 시스템 반영 사이클 walkthrough
  - 발견 건 전체에 대해 후속조치 이력 100% 추적
- **결함 판단**:
  - 발견했는데 후속조치 X → 결함 (KISA 결함사례 3번)
  - 후속조치 일부만 (예: 5건 발견 → 2건만 회수, 3건 유지 사유 X) → 결함
  - 발견 건 전체 후속조치 + 사유 기록 → OK

### Q4. 장기 미접속자 기준 — KISA 명시 있나? 6개월?
A. **KISA 명시 숫자 X. 결함사례에 "6개월 이상" 예시만. 회사 정책이 기준.**
- KISA 안내서: "장기 미접속자 계정 현황 및 삭제(또는 잠금)" — 숫자 X
- **KISA 결함사례 2번에 "6개월 이상 미접속 계정이 활성" 사례** = 6개월이 사실상 표준 시그널
- **업계 관행**:
  - 일반 시스템 6개월 ~ 1년 = 표준
  - 개인정보처리시스템 = 90일 권고 (안전성 확보조치 기준 옛 권고 영역)
  - 금융·고위험 = 30일 ~ 90일
- **검증 절차**:
  - 회사 정책 기준 미접속자 query (예: `last_login_date < CURRENT_DATE - 180`)
  - 결과 list vs 잠금·삭제 이력 대조
  - 회수 누락된 계정 = 결함
- **결함 판단**:
  - 정책에 미접속 기준 X → 결함 (KISA 결함사례 1번)
  - 정책 = 6개월인데 6개월 이상 미접속 활성 계정 다수 → 결함 (KISA 결함사례 2번 그대로)
  - 정책 = 1년 + 1년 미접속자 검토·잠금 적시 적용 → OK

### Q5. 한국 비금융 ITGC 실무, 2.5.6 어디까지 봐야 함?
A. **§3-A의 8개 거의 다 본다. ITGC APD 영역의 핵심·외부감사인 결함 빈출 영역. 무겁게.**
- **무겁게 보는 이유**:
  - 회계감사 ITGC = "권한 부여 → 사용 → 검토" 사이클이 통제 작동의 증명
  - 검토 사이클이 형식적이면 부여(2.5.1·2.5.5) 통제도 신뢰 X
  - 외부감사인 ITGC 검증에서 통상 1~2개 결함 발견 영역
- **빈출 결함 패턴 (양획 본업 시각)**:
  - 검토 정책은 있는데 사이클 X (KISA 결함사례 1번)
  - 6개월 이상 미접속자 다수 활성 (KISA 결함사례 2번)
  - 검토 발견 건 후속조치 X 또는 일부만 (KISA 결함사례 3번)
  - IT팀 단독 검토 + 현업 서명 X (형식적 검토)
  - 휴직·퇴직자 권한 회수 누락 (HR 변동 vs 시스템 계정 불일치 — 2.5.1 Q1과 묶음)
- **회계감사 시각 핵심**:
  - 정책 + 사이클 + 결과 + 시정 4단계 다 작동해야 통제 effective
  - 어느 한 단계 빠지면 전체 사이클 결함

## 관련 인증기준

## 관련 법규 / 표준
- 개인정보의 안전성 확보조치 기준 제5조 (접근권한 관리), 제8조 (접속기록 1년+ 보관)
- 회계감사기준서 315 (IT 위험 이해) — 권한 통제 사이클 검증 영역
- COBIT DSS05.04 (Manage User Identity and Logical Access) — 정기 권한 검토
- NIST SP 800-53 AC-2(7) (Account Management — Privileged User Accounts Review)
