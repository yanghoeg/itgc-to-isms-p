# 2.5.4 비밀번호 관리

## 1. ISMS 원본

### 인증기준
법적 요구사항, 외부 위협요인 등을 고려하여 **정보시스템 사용자 및 고객, 회원 등 정보주체(이용자)** 가 사용하는 **비밀번호 관리절차를 수립·이행**하여야 한다.

### 주요 확인사항 (KISA 안내서)
1. 정보시스템에 대한 안전한 사용자 비밀번호 관리절차 및 작성규칙을 수립·이행하고 있는가?
2. 정보주체(이용자)가 안전한 비밀번호를 이용할 수 있도록 비밀번호 작성규칙을 수립·이행하고 있는가?
3. 개인정보취급자 또는 정보주체의 인증수단을 안전하게 적용하고 관리하고 있는가?

### 관련 법규
- 개인정보 보호법 제29조 (안전조치의무)
- 개인정보의 안전성 확보조치 기준 제5조 (접근권한 관리)

### 비밀번호 작성규칙 (KISA 안내서, 시스템적 강제화 권고)

| 구분 | 내용 |
|---|---|
| **조합 규칙** | 영문·숫자·특수문자 중 **2종류 이상 조합 + 최소 8자리** / 문자만 구성 시 **최소 10자리** (숫자만 구성 X) |
| **변경주기** | 유효기간 설정 + 주기 변경. **단, 주기적 변경 여부·변경주기는 위험평가 결과로 자체 결정** |
| **추측 쉬운 비밀번호 제한** | 동일 문자 반복, 키보드 나란히 문자열, 일련번호·연속 숫자, 생일·전화번호 등 개인정보, ID와 비슷한 비밀번호 사용 제한 |
| **재사용 제한** | 변경 시 이전 비밀번호 재사용 제한 |

### 비밀번호 관리절차 (KISA 안내서 예시)
- 시스템 도입 시 초기/임시 비밀번호 변경 후 사용
- 비밀번호 입력·변경 시 마스킹 처리
- 종이·파일·모바일 기기 비밀번호 기록·저장 제한 (부득이한 경우 암호화)
- 침해사고 발생 또는 노출 의심 시 지체 없이 변경
- 비밀번호 분실 재설정 시 본인확인 절차
- 관리자 비밀번호 = 비밀등급에 준하여 관리

### 비밀번호 외 인증수단 (제5조제5항)
- 인증서·PIN·생체인식·보안토큰 등 = 탈취·도용 방지 보호대책 적용

### 결함사례 (KISA 안내서)
- **사례 1**: 정책에 비밀번호 생성규칙을 정해놓았으나 **일부 시스템에서 내부 지침과 상이한 비밀번호** 사용
- **사례 2**: 비밀번호 초기화 시 임시 비밀번호를 부여받고 **강제적으로 변경하도록 되어 있으나 실제로는 임시 비밀번호를 그대로 사용**
- **사례 3**: 비밀번호 변경주기를 정하고 이행하도록 했음에도 **변경하지 않고 그대로 사용**

## 3. IT감사에서는 이렇게

### 3-A. 한국 비금융 ITGC 실무 표준 (실제 검증 범위)

비밀번호 정책은 2.5.3 영역(실패횟수·세션 타임아웃)과 한 화면에 같이 나오므로 보통 **하나의 자동통제 캡처로 동시 검증**한다.

| ISMS 요구 | ITGC 테스트 절차 | 모집단 | 샘플 |
|---|---|---|---|
| 1. 비밀번호 작성규칙 | 시스템 설정값 캡처 (길이·복잡도·금지 패턴) | 전 시스템 | 자동통제 1건 샘플 |
| 2. 비밀번호 변경주기 | 시스템 설정값 캡처 (유효기간 일수, 회사 정책과 일치) | 전 시스템 | 자동통제 1건 샘플 |
| 3. 재사용 제한 | 시스템 설정값 캡처 (최근 N개 비밀번호 재사용 차단) | 전 시스템 | 자동통제 1건 샘플 |
| 4. 임시 비밀번호 강제 변경 | 신규 계정·재설정 계정에 첫 로그인 시 비밀번호 변경 강제 | 전 시스템 | 자동통제 1건 샘플 |
| 5. 정책 문서 | 회사 비밀번호 정책 문서 (entity-level) | 정책 1건 | 정책 1건 walkthrough |

### 3-B. ISMS-P가 추가로 요구하지만 ITGC 실무 범위 밖

| ISMS 요구 | 절차 | 어디 영역 |
|---|---|---|
| 정보주체(이용자) 비밀번호 작성규칙 | web 회원가입 화면의 비밀번호 정책 검증 | ISMS-P 인증심사 (B2C 서비스 한정) |
| 비밀번호 마스킹 처리 | 입력·변경 화면 캡처 | ISMS-P 인증심사 |
| 종이·파일·모바일 기록 제한 | 보안 정책 + 인터뷰 + 책상 점검 | **보안 영역** (sticky note 점검은 보안 진단) |
| 분실 재설정 시 본인확인 | 재설정 절차 walkthrough + 사례 1건 | ISMS-P 인증심사 |
| 비밀번호 외 인증수단 보호 (인증서·생체) | 인증수단별 보호대책 | ISMS-P 인증심사 |

## 4. Q&A

### Q1. 비밀번호 변경주기 90일 무조건 결함인가? KISA 입장은?
A. **KISA 2023.11 안내서가 변경주기를 위험평가 결과로 자체 결정으로 완화. 90일 주기 무조건 강제 X.**
- KISA 안내서 원문: "비밀번호 유효기간을 설정하여 주기적으로 변경(단, **주기적 변경 여부 및 변경주기는 위험평가 결과 등을 고려하여 자체적으로 결정**)"
- 배경: 미국 NIST SP 800-63B (2017~) 권고 반영. **잦은 비밀번호 변경 강제가 오히려 약한 비밀번호·sticky note 기록 유발**한다는 보안 연구 결과
- 회사 위험평가 결과로 "변경주기 X" 결정해도 OK. 단 충분한 보완통제 필요 (긴 비밀번호·MFA·이상 로그인 모니터링)
- **결함 판단**:
  - 회사 위험평가 + 정책 수립 + 정책 일관 적용 = OK
  - 위험평가 X + 변경주기 X + 보완통제 X = 결함
  - 정책상 90일인데 시스템 설정 안 되어 있음 = 결함 (정책-실제 일치성 부재)

### Q2. 정책상 8자리인데 시스템은 6자리 — 결함?
A. **결함. 통제 목적 달성 X (KISA 결함사례 1번 그대로). 단 미세 차이는 영향 평가.**
- 통제 목적 = 추측 어려운 비밀번호 강제. 6자리 = 무차별 대입 공격 1초 단위 가능 → 통제 목적 달성 X
- KISA 결함사례 1번: "일부 시스템에서 내부 지침과 상이한 비밀번호" → **즉시 결함**
- 미세 차이 (정책 9자리·시스템 8자리) = 통제 목적 달성 → **결함 X** (2.5.3 Q2 기준과 일관)
- **차이 판단 기준**: 통제 목적 달성 여부 — 8자리 미만은 결함, 8자리 이상이면 미세 차이 immaterial

### Q3. 임시 비밀번호 강제 변경 — 어디까지 검증?
A. **KISA 결함사례 2번 명시 영역. 신규·재설정 계정 1건 walkthrough로 검증.**
- 검증 절차:
  - 신규 계정 발급 시 임시 비밀번호 부여 → 첫 로그인 시 변경 강제 (시스템 설정 캡처 + 1건 walkthrough)
  - 비밀번호 분실 재설정도 동일
- 자동통제이므로 1건 샘플 + ITGC 일반통제 의존
- **결함 판단**:
  - 강제 변경 설정 자체 X → 결함 (KISA 결함사례 2번)
  - 정책상 강제이지만 "임시 비밀번호 그대로 사용 중" 사례 발견 → 결함 (정책-실제 불일치)
  - 강제 변경 설정 + 1건 사례 정상 → OK

### Q4. sticky note·메신저 비밀번호 공유 — ITGC 결함?
A. **ITGC 영역 X. 보안 진단·인사 영역. 다만 다른 통제와 묶여서 결함 가능.**
- KISA 안내서: "종이·파일·모바일 기기 등 비밀번호 기록·저장 제한"
- ITGC 시각 = 정책 수립·보안 서약·인식 교육까지가 검증 범위. **sticky note 책상 점검은 보안 진단·내부감사 영역**
- 양획 본업 시각: 책상 점검은 정보보안팀·내부감사가 함. ITGC는 정책 + 절차 + 적용 사례 인터뷰까지
- **연결 통제** (2.5.2 Q5 본인 사용 보장):
  - 비밀번호 공유 금지 정책 (entity-level) → 2.1 정책
  - 보안 서약서 작성 → 2.2.3
  - 인식 교육 → 2.2.4
  - 위반 시 조치 → 2.2.6
- **결함 판단**:
  - ITGC 단독으로는 비밀번호 공유 자체를 결함으로 잡지 X
  - 다만 정책·서약·교육 자체가 부재 → entity-level 결함 가능
  - 인터뷰에서 비밀번호 공유 정황 발견 + 정책·서약 부재 → 묶어서 결함

### Q5. 한국 비금융 ITGC 실무, 2.5.4 어디까지 봐야 함?
A. **§3-A의 5개 (작성규칙·변경주기·재사용 제한·임시 비밀번호 강제 변경·정책 문서). 2.5.3과 한 캡처로 동시 검증.**
- 한 화면 (AD GPO·Linux PAM·DB 사용자 정책·web 인증 설정)에 비밀번호 작성규칙·변경주기·재사용 제한이 다 같이 나옴 → 2.5.3 (실패횟수·세션 타임아웃)과 묶어서 자동통제 1건 캡처
- 임시 비밀번호 강제 변경 = 별도 walkthrough 1건 (신규 계정 발급 사례)
- 정책 문서 = entity-level 1건 검증
- **나머지 (B2C 회원가입 정책·sticky note 점검·인증서 보호 등) = ISMS-P 인증심사·보안 영역.** ITGC에서 다루지 않아도 무방
- 회계감사 ITGC와 정보보안 영역의 영역 분담 결과

## 관련 인증기준
- 2.5.2 사용자 식별 (Q5 비밀번호 공유 다층 통제와 연결)
- 2.5.3 사용자 인증 (실패횟수·세션 타임아웃과 한 캡처)
- 2.5.5 특수 계정 및 권한 관리 (관리자 비밀번호 비밀등급 관리)
- 2.7.1 암호정책 (비밀번호 일방향 암호화 — 안전성 확보조치 기준 제7조제1항)
- 2.2.3 보안 서약 / 2.2.4 인식 교육 (비밀번호 공유 금지 entity-level 통제)

## 관련 법규 / 표준
- 개인정보의 안전성 확보조치 기준 제5조 (접근권한 관리), 제7조제1항 (비밀번호 일방향 암호화)
- NIST SP 800-63B (Authentication and Lifecycle Management) — 변경주기 강제 완화 권고 (2017~)
- KISA 2023.11 안내서 변경주기 자율화 = NIST 권고 반영
